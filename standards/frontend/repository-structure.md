# Frontend Repository Structure Spec

## Purpose

The Nexus web client lives in `web/` at the repository root, alongside the Python
API in `app/`. It is a separate build and deployment artifact (see
`../../docs/adr/0012-web-client-and-separate-deployments.md`). This document is the
structural contract for that client, mirroring the discipline of
`backend/repository-structure.md`.

Do not add new top-level folders inside `web/src` or introduce generic `utils`/
`helpers` dumping grounds unless this spec is updated first.

## Stack

| Layer | Technology |
| --- | --- |
| Runtime | Bun (dev/build); the built bundle is static assets. |
| Framework | React + TypeScript |
| Routing | TanStack Router (file-based, `web/src/routes`) |
| Server state | TanStack Query |
| Styling | Tailwind CSS v4 with tokens from `DESIGN.md` |
| Build | Vite |

## Required Layout

```text
web/
  package.json
  vite.config.ts
  tsconfig.json
  index.html
  .env.example              # VITE_API_URL and other public build vars
  src/
    main.tsx                # app bootstrap
    routeTree.gen.ts        # generated by TanStack Router
    styles/
      globals.css           # Tailwind entry + @theme tokens from DESIGN.md
    lib/
      queryClient.ts        # TanStack Query client
    api/
      client.ts             # typed fetch, auth header, problem+json handling, refresh
      types.ts              # domain enums/DTOs mirroring the API contract
      <resource>.ts         # one module per API area (auth, memory, search, ...)
    auth/
      authStore.tsx         # session/token state and route guard helpers
    components/
      ui/                   # design-system primitives (Button, Card, Badge, ...)
      <domain components>   # MemoryCard, FilterBar, EvidenceList, ...
    routes/                 # file-based routes; one file per view
```

## Layer Responsibilities

| Path | Responsibility | Must not do |
| --- | --- | --- |
| `routes/` | Route definitions, loaders, page composition. | Raw fetch logic or design primitives. |
| `api/` | All network access and DTO typing. | React/rendering concerns. |
| `auth/` | Session/token lifecycle and guards. | Business/domain rendering. |
| `components/ui/` | Reusable design-system primitives. | API calls or route knowledge. |
| `components/` | Domain widgets composed from `ui/`. | Owning global app state. |
| `styles/` | Tailwind config and design tokens. | Component logic. |

## Rules

| Rule | Requirement |
| --- | --- |
| API-only | The client uses the `/v1` API exclusively; never PostgreSQL, indexes, or vector stores. |
| Respect review | Treat shared memory as possibly `pending_review`/`needs_review`; surface warnings. |
| Error branching | Branch on HTTP status and problem `code`, not on human `detail`. |
| Tokens | Keep the access token in memory; store only the refresh token in browser storage. |
| Design tokens | Colors, typography, radii, spacing, and shadows come from `DESIGN.md`; do not invent new hues. |
| No API base hardcoding | The API base URL is `VITE_API_URL` (build-time), never hardcoded per call. |
